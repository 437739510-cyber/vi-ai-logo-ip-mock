## 1. 产品背景与目标

### 1.1 背景

大量中小企业在品牌建设初期需要一套专业的 VI（视觉识别）手册，但委托专业设计公司成本高、周期长，且沟通成本大。同时市场上缺乏面向供应商侧的 AI 辅助 VI 生成工具。

### 1.2 产品定位

**B2B 代运营服务产品**。客户通过极简界面提交品牌素材与需求，我方运营团队借助 AI 辅助完成 VI 手册的设计与交付。产品的核心用户并非终端企业，而是我方内部的设计与运营团队。

### 1.3 商业目标

- 降低 VI 手册设计的单客交付成本，缩短交付周期
- 通过 AI 辅助实现规模化接单
- 积累品牌资产库，形成行业壁垒

      │        │        │        │
┌─────▼──┐ ┌──▼────┐ ┌──▼───┐ ┌──▼─────┐
│PostgreSQL│ │DeepSeek│ │ 本地  │ │public/  │

└─────────┘ └───────┘ └──────┘ └─────────┘

```
客户提交需求 ──→ 销售跟踪确认 ──→ 设计师使用AI生成方案
                       ↓              设计师编辑精修 ──→ 内部审核
                       ↓              导出交付 ──→ 通知客户
```

## 5. 客户端需求详情

### 5.1 首页 (Homepage)

首页向潜在客户传递服务价值、建立信任、引导完成需求提交。包含 Hero 区域、服务流程、案例展示、服务优势、FAQ、底部 CTA。

### 5.2 咨询留资 (Consultation)

客户填写需求信息并上传品牌素材的表单页面。字段包括：联系人姓名、公司名称、联系电话、微信、所属行业、预算范围、需求描述。文件上传支持：LOGO（PNG/JPG/SVG，最多5个）、IP公仔（PNG/JPG/GIF，最多10个）、参考VI手册（PDF，最多1个）。

### 5.3 确认页 (Confirm)

提交成功后展示项目编号，引导客户使用编号查询进度。

### 5.4 进度查询 (Progress)

输入项目编号+手机号查询当前进度，展示时间线。

## 6. 管理端需求详情

### 6.1 登录 (Login)

管理员登录页面（简单密码验证或后续接入 OAuth）。

### 6.2 工作台 (Dashboard)

项目统计看板，展示待处理事项列表。

### 6.3 项目列表 (Projects)

按状态筛选、搜索项目编号。每行显示：编号、状态、负责人、创建时间、操作。

### 6.4 项目详情 (Project Detail)

展示客户信息、品牌素材、AI 生成方案状态、VI 手册状态。
- AI 方案：已生成/未生成状态标识 + 查看方案按钮 + 重新生成（含反馈输入）
- VI 手册：已生成/未生成状态 + 编辑/预览/导出按钮
- AI 逐页生成：调用通义万相的入口
- 项目时间线

### 6.5 AI 方案生成 (Generate)

选择参考模式（强/弱/不参考），调用 DeepSeek 生成多套方案。每套方案以 SVG 预览卡片展示（主/辅/强调色色块 + 字体搭配 + 风格描述）。可收藏/取消收藏。

### 6.6 VI 编辑器 (Editor)

色板编辑（HEX/CYMK/RGB 输入或取色器）、字体选择、版式调整。

### 6.7 手册预览 (Preview)

翻页预览完整手册（封面、Logo规范、色彩系统、字体规范、辅助图形、应用系统等 9 页）。

### 6.8 AI 逐页生成 (Manual Pages)

调用通义万相 API 逐页生成 11 页手册内容：
- 每次调用前弹出费用确认弹窗
- 生成后展示预览
- 不满意可输入修改意见重新生成
- 满意后确认进入下一页
- 全部完成后可预览完整手册或导出

### 6.9 导出 (Export)

选择导出格式（PDF），生成可交付文件。

## 9. 非功能需求

### 9.1 性能

- 客户端首页首屏加载 ≤ 2s（SSR）
- 文件上传（20MB）≤ 30s
- AI 方案生成（含参考手册分析）≤ 30s，SSE 流式反馈即时进度
- 管理端列表加载 ≤ 1s

### 9.2 安全

- 所有 API Key 通过环境变量注入，严禁硬编码
- 文件上传限制格式、大小，禁止可执行文件
- 管理端要求登录态
- 客户端无需登录

### 9.3 兼容性

- Chrome / Edge / Safari 最新版
- 客户端核心流程（表单、上传、进度查询）在手机可用
- 管理端适配 1280px+ 桌面

### 9.4 可维护性

- TypeScript strict mode
- 组件设计遵循原子化，通用组件抽离
- 业务逻辑复杂处加注释
- 通过环境变量 `NEXT_PUBLIC_USE_MOCK` 控制 Mock/真实 API 切换

## 10. 技术栈

### 10.1 前端

| 类别 | 技术选型 | 选型理由 |
|---|---|---|
| 框架 | **Next.js 15** (React 19, App Router) | SSR 提升首页 SEO；双端共用一套代码通过路由权限区分 |
| 语言 | TypeScript (strict mode) | 类型安全 |
| 样式 | Tailwind CSS 4 | 快速开发，一致的 Design Token |
| UI 组件 | shadcn/ui | 基于 Radix，可访问性好，视觉专业 |
| 状态管理 | Zustand | 轻量，适合管理端全局状态 |
| 表单 | React Hook Form + Zod | 表单校验，类型推导 |
| 文件上传 | react-dropzone | 拖拽上传体验 |
| 动画 | Framer Motion | 页面过渡、卡片动画 |
| 取色器 | react-colorful | 色板编辑 |

### 10.2 AI / 后端

| 类别 | 技术选型 | 说明 |
|---|---|---|
| 后端框架 | **Next.js API Routes** | 全栈一体化，无需独立后端服务 |
| AI 方案推理 | DeepSeek API | 方案生成、参考手册分析 |
| AI 图像生成 | 通义万相 API | 手册页面图片生成 |
| 数据库 | Supabase (可选) + 本地 JSON 备份 | 轻量部署，JSON 文件作为主要存储 |
| 流式传输 | SSE (Server-Sent Events) | AI 生成进度实时推送 |

### 10.3 DevOps

| 类别 | 选型 |
|---|---|
| 前端托管 | Vercel |
| 域名/CDN | Cloudflare |
